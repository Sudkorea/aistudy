우체국 금융 대규모 고객 데이터 기반 추천 시스템 리팩터링 및 고도화 프로젝트 (2025.02 ~ 진행 중)

활용 기술 및 도구

- 추천 알고리즘: ALS, LightGCN, Two-Tower
    
- 데이터 처리: Python (Pandas, NumPy, SciPy), Spark, Hive
    
- 실험 및 평가 지표 설계: Precision@K, Recall@K, Item Coverage, A/B Test 설계
    
- 최적화: GPU/메모리 절감 및 전처리 파이프라인 리팩터링
    

본인의 역할

- 추천 모델 학습 및 실험 담당
	- 팀 내 모델 실험 및 평가 지표 설계 전담
	- 레거시 시스템 구조 분석 및 개선 전략 주도
	- 다양한 알고리즘 실험을 통해 성능·자원 효율성·확장성을 종합적으로 고려

- 학습 내용 전달 담당
	- 학습한 내용을 알기 쉬운 방식으로 바꾸어 팀원들에게 전달
	- 담당 직원 및 일반 직원 대상 프로젝트 내용 및 진행 상황 발표

진행 방식

-  EDA: 프로젝트 담당자와 공동으로 진행하여, 금융 데이터 및 우체국 금융 데이터만의 특징 파악, 기존 데이터 전처리 과정에서의 문제점 다수 발견
	
- 리팩터링: 하드코딩 제거와 모듈화 → RAM 40GB → 20GB, GPU 7GB → 2GB 절감
    
- 데이터 전처리: 수치형 변수 log1p+표준화, 범주형 변수 One-Hot/Binary Encoding
    
- 클러스터링 개선: K-Means k 값 실험적 조정 + 동적 학습 파이프라인 도입
    
- 평가지표 확장: hit 지표만 있던 기존 구조에 Precision·Recall·Item Coverage 추가 및 샘플링 오류 제거 쿼리 작성
    

문제 해결 경험

- 데이터 정규화 관련
	- 레거시 코드의 클러스터링 시 연령대에 지나친 가중치 부여, 특정 연령대에 몰린 데이터 특성 상 유의미하지 않은 클러스터링 방식 -> 로직 수정 시 recall 1.5배 증가하는 것을 확인
	- 금융 데이터 특성상 잔액이 0에 지나치게 많이 분포하나, 이를 스케일링하지 않는 문제 발견 -> 표준화 시 recall 1.3배 증가 확인
	
- 추천 로직 오류: 모든 고객에게 동일한 추천 제공 문제 발견 → 로직 전면 수정
    
- 추천 편향 문제: 특정 고객 집단에 한정된 정확도 발생 → 가중 샘플링 실험으로 개선 시도
    
- 모델 성능 최적화: ALS, LightGCN, Two-Tower 모델을 실험하며, 대규모 데이터 제약 속에서 최적 모델 탐색 주도
	
- 현업에서 나타난 메모리 초과 문제 실시간 트러블슈팅 경험
    

성과 및 향후 계획

- ALS 성능 대폭 개선: Precision@5 2% → 16%, Recall@5 8.4% → 64.3%
    
- LightGCN/Two-Tower: 사용자-상품 관계 반영 모델에서 기존 대비 우수한 성능 확인
    
    - 다만, 클러스터링 적용으로 인한 성능 차이 분석 필요 → 메모리 제약 고려한 경량화 코드 재조정 진행 예정
        
- 향후 계획:
    
    - A/B 실시간 지표 기반 평가 체계 구축
        
    - 도메인 기반 클러스터링 그룹 재설정 (예: 교차추천 고객, 자산 규모별) 및 그룹별 맞춤 모델 탐색
        
    - TDA(Topological Data Analysis) 개념 정리 및 적용 가능성 실험 예정
